# XL-lite 功能缺失与建设计划

## 1. 当前项目定位

XL-lite 当前已经具备全栈后台管理系统的基础骨架：Vue 3 管理端、NestJS 服务端、JWT 双 Token 登录、图形验证码、动态菜单、菜单管理、个人中心、头像上传、Swagger 文档、PostgreSQL、Redis、BullMQ 基础设施，以及 RBAC 相关的用户、角色、权限、菜单实体和接口雏形。

但从实际代码覆盖看，项目还没有达到 README 中“完整 RBAC 权限管理后台”的状态。目前更接近“后台模板底座 + 登录鉴权 + 菜单管理 + RBAC 后端雏形”。

## 2. 已具备功能

### 2.1 认证与用户基础能力

- 账号密码登录。
- 图形验证码登录校验。
- Access Token + Refresh Token 刷新机制。
- 退出登录。
- 获取当前用户信息。
- 当前用户资料修改。
- 当前用户密码修改。
- 当前用户头像上传与更新。

### 2.2 菜单与路由能力

- 后端菜单实体、菜单 CRUD 接口。
- 前端菜单管理页面。
- 后端返回菜单树。
- 前端基于后端菜单动态生成路由。
- 菜单权限码基础字段。

### 2.3 RBAC 基础结构

- 用户实体。
- 角色实体。
- 权限实体。
- 用户与角色关联。
- 角色与权限关联。
- 超级管理员种子数据。
- 获取当前用户权限码。

### 2.4 基础设施

- NestJS 模块化结构。
- TypeORM 数据库访问。
- PostgreSQL 支持。
- Redis 支持。
- BullMQ 队列基础模块。
- 文件上传基础接口。
- Swagger API 文档。
- 统一响应拦截器。
- 全局认证与权限守卫。

## 3. 当前缺失功能

### 3.1 系统管理前端页面缺失

当前前端 `system` 目录只实现了菜单管理页面，系统管理的核心页面不完整。

缺失功能：

- 用户管理页面。
- 角色管理页面。
- 权限管理页面。
- 用户分配角色交互。
- 角色分配权限交互。
- 用户状态启用/禁用操作。
- 用户重置密码操作。
- 用户搜索、筛选、分页。
- 角色搜索、筛选、分页。

影响：

- 后端虽有用户、角色、权限接口，但管理端无法完成完整 RBAC 配置。
- 种子菜单中的用户管理、角色管理仍处于注释状态，实际系统菜单没有开放这些功能。

### 3.2 RBAC 后端服务未完全实现

后端用户、角色、权限控制器已存在，但部分服务层仍是占位或简单实现。

缺失功能：

- 用户列表真实查询，目前 `UserService.findAll()` 仍是占位字符串。
- 角色列表真实查询，目前 `RoleService.findAll()` 仍是占位字符串。
- 用户分页查询。
- 用户按用户名、邮箱、状态、角色筛选。
- 角色分页查询。
- 角色按名称、编码、状态筛选。
- 创建用户时的用户名/邮箱唯一性校验。
- 更新用户时同步维护角色关联。
- 更新角色时同步维护权限关联。
- 删除用户前的约束校验。
- 删除角色前检查是否仍被用户使用。
- 删除权限前检查是否仍被角色使用。
- 权限码和菜单按钮权限的完整关联策略。

影响：

- RBAC 关系可以初始化，但还不能稳定支撑日常后台管理。
- 简单 `update()` 无法维护多对多关系，后续容易出现角色/权限关联不同步。

### 3.3 前端 API 封装缺失

当前前端核心 API 只覆盖 `auth`、`menu`、当前用户资料相关接口。

缺失功能：

- 管理端用户 CRUD API 封装。
- 角色 CRUD API 封装。
- 权限 CRUD API 封装。
- 用户分页查询类型定义。
- 角色分页查询类型定义。
- 权限列表类型定义。
- 用户分配角色请求类型。
- 角色分配权限请求类型。

影响：

- 前端页面即使开发，也缺少统一 API 层支撑。
- 类型边界不清晰，后续容易出现接口字段不一致。

### 3.4 认证扩展页面存在但未接业务

前端路由中存在注册、验证码登录、二维码登录、忘记密码页面，但登录页入口关闭，后端也没有对应业务接口。

缺失功能：

- 注册接口。
- 注册页面业务接入。
- 忘记密码接口。
- 重置密码流程。
- 邮箱验证码或短信验证码登录。
- 二维码登录后端能力。
- 第三方登录能力。
- 登录失败次数限制。
- 账号锁定策略。
- 登录日志记录。

建议：

- 如果短期不建设这些能力，应从路由或文档中标记为模板能力，避免误导。
- 如果需要对外开放注册，应优先补账号安全策略和验证码/邮箱验证。

### 3.5 任务队列能力仍是骨架

BullMQ 队列、Worker、Trigger、Strategy Registry 已存在，但业务任务服务仍为空实现。

缺失功能：

- 任务定义管理。
- 任务触发接口。
- 任务列表查询。
- 任务执行日志。
- 任务状态查看。
- 失败任务重试。
- 定时任务或延迟任务配置。
- 前端任务管理页面。
- 至少一个真实业务任务示例。

影响：

- README 中“任务队列”能力目前只能算基础设施，尚不是可用功能模块。

### 3.6 文件管理能力不足

当前只有基础上传接口，缺少文件资产管理闭环。

缺失功能：

- 文件列表。
- 文件删除。
- 文件元数据入库。
- 文件大小限制配置。
- 文件类型白名单。
- 图片预览。
- 上传失败提示规范。
- 对象存储扩展，如 OSS、COS、S3。
- 文件访问权限控制。
- 前端文件管理页面。

影响：

- 头像上传可用，但不能作为完整文件管理模块使用。

### 3.7 仪表盘未接真实数据

前端存在工作台和分析页，但更像模板展示页面，没有看到对应后端统计接口。

缺失功能：

- 用户数量统计。
- 角色数量统计。
- 菜单数量统计。
- 今日登录统计。
- 最近操作记录。
- 系统运行状态。
- 任务执行概览。
- 后端 dashboard 统计接口。

影响：

- 首页不能反映真实系统状态，生产使用价值有限。

### 3.8 日志与审计能力缺失

后台管理系统通常需要可追踪、可审计的系统行为记录，目前没有看到完整实现。

缺失功能：

- 操作日志。
- 登录日志。
- 异常日志查看。
- API 访问日志。
- 日志分页查询。
- 日志按用户、时间、模块、操作类型筛选。
- 敏感操作审计。

影响：

- 不利于问题排查、安全审计和后台责任追踪。

### 3.9 系统配置与基础字典缺失

当前项目缺少常见后台基础配置模块。

缺失功能：

- 参数配置。
- 字典类型管理。
- 字典数据管理。
- 系统公告。
- 站内通知。
- 组织架构。
- 部门管理。
- 岗位管理。
- 数据权限。

建议：

- 轻量模板可以不一次性全部实现，但至少应规划字典、参数配置、操作日志三个基础模块。

### 3.10 文档与实际实现不一致

当前 README、后端 README 与实际代码存在不一致。

问题：

- 根 README 写前端 UI 库为 PrimeVue，但当前项目实际使用 vben 体系和 TDesign。
- Swagger 地址描述不统一。
- 后端 README 残留 “XL Video Host / 视频模块” 内容，但当前代码未见视频模块。
- README 宣称完整 RBAC，但前端页面和后端服务尚未完整闭环。

影响：

- 新开发者容易按错误文档理解项目。
- 项目能力边界不清晰。

## 5. 推荐优先级路线图

### P0：补齐 RBAC 闭环

目标：让系统具备真正可用的用户、角色、权限、菜单管理能力。

任务：

1. 实现用户列表、分页、搜索、创建、编辑、删除、状态切换、重置密码。
2. 实现用户与角色关联的新增和更新。
3. 实现角色列表、分页、搜索、创建、编辑、删除、状态切换。
4. 实现角色与权限关联的新增和更新。
5. 实现权限管理或权限树数据源。
6. 新增前端用户管理页面。
7. 新增前端角色管理页面。
8. 新增前端权限管理页面或授权弹窗。
9. 打开系统管理中的用户、角色菜单。
10. 补齐前端 `user`、`role`、`permission` API 封装。

验收标准：

- 管理员可以在页面创建用户。
- 管理员可以给用户分配角色。
- 管理员可以创建角色并分配权限。
- 不同角色登录后看到不同菜单和权限码。
- 用户禁用后不能登录。
- 删除角色时，如果仍有用户使用，应给出明确错误。

### P1：补齐后台基础运维能力

目标：提升项目作为后台模板的生产可用性。

任务：

1. 实现登录日志。
2. 实现操作日志。
3. 实现 dashboard 真实统计接口。
4. 接入首页真实统计数据。
5. 完善文件上传校验。
6. 增加文件列表和删除能力。
7. 增加基础系统配置。
8. 增加字典管理。

验收标准：

- 登录成功/失败均有日志。
- 用户、角色、菜单等关键操作有操作日志。
- 首页展示真实用户数、角色数、登录数等数据。
- 文件上传有大小和类型限制。
- 管理员可以查看和删除上传文件。

### P2：扩展认证、安全与任务能力

目标：补齐安全策略和队列任务的真实业务价值。

任务：

1. 登录失败次数限制。
2. 账号锁定策略。
3. 忘记密码流程。
4. 注册流程，按产品需要决定是否开放。
5. 邮箱验证码或短信验证码能力。
6. 任务队列增加真实业务任务。
7. 任务执行日志。
8. 任务重试和失败查看。
9. 任务管理前端页面。

验收标准：

- 连续登录失败会触发限制或锁定。
- 用户可以通过安全流程重置密码。
- 队列任务可以被触发、执行、记录和重试。
- 管理员可以在页面查看任务执行状态。

### P3：增强型后台能力

目标：根据业务方向决定是否继续扩展为更完整的中后台平台。

可选任务：

1. 部门管理。
2. 岗位管理。
3. 数据权限。
4. 站内通知。
5. 系统公告。
6. 在线用户管理。
7. API 权限资源自动扫描。
8. 对象存储支持。
9. 多租户能力。

验收标准：

- 根据具体业务需求单独定义。
- 不建议在 P0/P1 未完成前过早展开。

## 7. 后续开发建议

### 7.1 先完成最小可用后台

建议先集中完成 P0，不要过早展开注册、二维码登录、第三方登录、多租户等扩展能力。当前项目最大的短板是系统管理闭环不完整，优先补齐用户、角色、权限、菜单之间的真实管理能力。

### 7.2 后端先于前端明确数据结构

用户、角色、权限相关接口建议先统一分页响应格式、查询 DTO、返回 DTO，再开发前端页面。这样可以避免前端表格字段反复调整。

建议统一列表响应结构：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "pageSize": 10
}
```

### 7.3 多对多关系不要用简单 update

用户-角色、角色-权限属于多对多关系，新增和更新时应显式查询关联实体并通过 `save()` 维护关系，不建议只调用 `repository.update()`。

### 7.4 文档应跟随功能同步更新

每完成一个模块，需要同步更新 README 或新增模块文档。尤其是默认账号、接口地址、菜单权限、启动方式、Swagger 地址等内容，应保持准确。

3. 后端：完善角色查询、新增、编辑、删除、状态切换、权限分配。
4. 后端：完善权限查询、新增、编辑、删除。
5. 后端：补充删除前约束校验。
6. 前端：新增用户管理 API 封装。
7. 前端：新增角色管理 API 封装。
8. 前端：新增权限管理 API 封装。
9. 前端：开发用户管理页面。
10. 前端：开发角色管理页面。
11. 前端：开发权限管理页面或角色授权弹窗。
12. 前端：打开系统管理菜单中的用户管理和角色管理。
13. 测试：验证不同角色登录后的菜单与权限码。

- 不同角色登录后权限码不同。
- 禁用用户无法登录。
- 删除正在使用的角色会被阻止。
- 删除正在使用的权限会被阻止。
